Code Structure
core/
The framework heart. Contains: agent.py (ReAct), chain.py (Sequential/Parallel/Router), orchestrator.py (multi-agent coordinator), memory.py (sliding window), security.py, providers/, tools/, knowledge/.
master/
Control plane. api/ holds the FastAPI app, routes, and WebSocket manager. frontend/ is the React Vite app served statically. Everything the user touches.
node/
worker.py connects to Redis, subscribes to task channels, instantiates an Orchestrator, executes agent tasks, and publishes results back. One file, one responsibility.
config/
agents.yaml defines every agent: name, provider, model, system prompt, tools, privilege level, chain membership. system.yaml holds defaults. No code changes needed to reconfigure agents.
core/agent.py
The ReAct Agent Loop
ReAct = Reason + Act. Each agent iterates: think about what to do → call a tool → observe the result → update memory → repeat until final answer or max iterations reached.
Example Task: "Summarize Q3 sales from the Excel file and compare to last year's PDF report"
Step 1/7
1
📥 User Input Received
The task is received and added to the agent's memory (sliding window, keeps last N turns).
memory.add("user", "Summarize Q3 sales from Excel, compare to last year PDF")
2
🧠 LLM Reasons → Decides Tool Call
The LLM sees memory + system prompt. It reasons: "I need to read the Excel file first." It emits a structured tool call.
{"tool": "read_excel", "args": {"path": "workspace/q3_sales.xlsx"}}
3
⚙️ Tool Dispatcher Executes
security.py checks privilege (STANDARD ≥ READ_ONLY ✓) and path sandbox (workspace/ ✓). Tool runs, returns sheet data.
tool_result = {"Q3 Total": 4_200_000, "Best Product": "Widget X", rows: [...]}
4
📝 Result Appended to Memory
Tool result goes into memory as a tool-role message. The loop continues — now the LLM knows the Excel data.
memory.add("tool", tool_result) # injected into next LLM call
5
🔁 Loop Iteration 2 — Read PDF
LLM now asks for the second piece of data: last year's PDF report. Another tool call, same dispatch cycle.
{"tool": "read_pdf", "args": {"path": "workspace/q3_2023_report.pdf"}}
6
🔍 Secret Redaction Applied
Before any output reaches the user or other agents, security.py runs regex redaction — API keys, tokens, passwords become [REDACTED].
output = redact_secrets(llm_response) # e.g. "sk-abc..." → "[REDACTED]"
7
✅ Final Answer Streamed
LLM has all data, produces a plain-text final answer. No more tool calls. Result streamed via WebSocket/SSE to the React frontend.
"Q3 2024 sales totalled €4.2M (+18% vs Q3 2023 €3.56M). Top performer: Widget X..."
core/memory.py
Sliding Window Memory
What it does
Keeps the last N conversation turns in memory. When the window is full, the oldest message is dropped. This keeps the LLM context within token limits without losing recent context.
Why not infinite?
Every LLM has a context window limit (e.g. 8K, 32K, 128K tokens). Sending the full history every time is expensive. Sliding window balances recency vs. cost.
Message roles
Memory stores messages by role: system (agent persona), user (task), assistant (LLM output), tool (tool results). Each role shapes LLM behavior.

core/orchestrator.py
Orchestrator
Role
Lives inside each node worker. Receives a task from Redis, looks up which chain/agent to invoke from config, executes it, and publishes the result back to the results channel.
Config-driven
No code changes to reconfigure agents. Just update agents.yaml: add a new agent, change its model, swap its chain membership. The orchestrator reads it at startup.
Fault isolation
If one node crashes, tasks requeue (or time out). Other nodes keep running. The master's WebSocket manager handles graceful disconnects and reconnects.

Why RAG vs Fine-tuning?
✅ RAG Advantages
No model retraining. Update knowledge by uploading new files. Works with any LLM provider. Transparent — you can see which chunks influenced the answer. Cheap.
⚠️ RAG Limitations
Only as good as the retrieval step. If the right chunk isn't in the top-N results, the LLM won't see it. Chunk size/overlap tuning matters a lot for quality.
🔧 This Implementation
1000-char chunks with 200-char overlap is a pragmatic default. The overlap ensures sentences split across chunk boundaries aren't lost. ChromaDB handles embedding + retrieval.

Privilege Levels
Tool Access Control
Level 0 — READ_ONLY
Can only read: files, web pages, Excel, Word, PDF, CSV, knowledge base. Zero write or execution capability. Safe for untrusted inputs.
Level 1 — STANDARD
Adds write tools: write_file, http_request (POST), write_excel/word/csv. Still no code execution. Good for productivity agents.
Level 2 — ELEVATED
Unlocks execute_python and execute_shell. Reserved for trusted automation agents. Path sandbox still applies.
Level 3 — ADMIN
Full unrestricted access. System management tasks only. Never assign to user-facing agents.
core/security.py
6-Layer Security Stack
Every HTTP request passes through sequential security layers. Click "Reveal Layers" to see them appear one by one, simulating a request travelling through the stack.
1
Rate Limiter
token-bucket per IP · 60 req/60s default · configurable via RATE_LIMIT_* env vars
→ 429 Too Many Requests
2
API Key Auth
X-API-Key header · HMAC constant-time compare · disabled if MASTER_API_KEYS not set (dev mode)
→ 401 Unauthorized
3
Node Secret Auth
X-Node-Secret header · only for node registration endpoints · prevents rogue node injection
→ 403 Forbidden
4
Privilege Level Gate
per-agent privilege (0–3) checked against each tool's min_privilege before execution
→ PermissionError
5
Path Sandbox
resolves symlinks · checks all file paths against ALLOWED_DIRS whitelist · no traversal
→ PermissionError
6
Secret Redactor
regex strips API keys, tokens, passwords from all LLM output before it leaves the system
→ [REDACTED]
Why each layer matters
HMAC Constant-Time Compare
Regular string comparison leaks timing information — an attacker can guess keys character by character by measuring response time. HMAC compare takes the same time regardless of where strings differ.
Symlink Resolution
A path like workspace/../../etc/passwd could escape the sandbox. By resolving symlinks and canonicalizing before checking, traversal attacks are neutralized.
Secret Redaction
LLMs sometimes echo back secrets from their context (API keys in env vars, database passwords in config files). The redactor ensures even if a secret enters the prompt, it never leaves in the response.
core/providers/
LLM Providers
Each agent independently chooses its LLM provider. Mix and match on the same node — run one agent on local Ollama, another on GPT-4o, another on Claude.
🦙 Ollama
FREE · Local
Runs models entirely on-device. No API key, no internet, no cost, maximum privacy. Best for sensitive data or offline environments.
llama3.2 mistral codellama phi3
⚡ Groq
Free Tier · Cloud
Ultra-fast LPU inference. Free tier is generous. Best for latency-sensitive tasks where you want cloud speed without per-token cost.
llama3-70b mixtral-8x7b gemma-7b
🟢 OpenAI
Paid · Cloud
GPT-4o. Highest general capability. Best for complex reasoning, code generation, and when quality is more important than cost.
gpt-4o gpt-4o-mini gpt-3.5-turbo
⬡ Anthropic
Paid · Cloud
Claude family. Best for long-context analysis, nuanced instructions, and tasks requiring careful reasoning over large documents.
claude-opus-4 claude-sonnet-4 claude-haiku
How to choose
Privacy-first?
→ Ollama. Data never leaves your machine. Slower, needs GPU for good performance, but fully sovereign. Use llama3.2 or mistral.
Speed matters?
→ Groq. Their LPU hardware is 10–20× faster than typical GPU inference. Free tier covers most dev workloads.
Max quality?
→ OpenAI GPT-4o or Anthropic Claude. For tasks like complex code review, long document analysis, or nuanced decision support.
Mixed fleet?
→ Use a sequential chain: Ollama agent does initial research (free), OpenAI agent does final synthesis (paid but minimal tokens).
